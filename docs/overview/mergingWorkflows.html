People use a variety of workflows when merging plugins. Below is a breakdown of these workflows including their pros and cons. These workflows didn't really have names, but I've assigned descriptive names to them here for the purpose of this comparison.

Disable Plugins Workflow

This is the default workflow in zMerge. This workflow involves disabling plugins in the plugins.txt file while leaving the merged mods installed. Disabled plugins do not count towards the 255 plugin limit.

Pros:
  • Allows you to rebuild merges in zMerge at any time without any extra steps.
  • Can be used with any mod manager.
  • zMerge has an integration which disables plugins after they are merged. This integration is enabled by default.
Cons:
  • Clutters your load order a little bit with disabled plugins.

Hide Plugins Workflow

This workflow was mostly used by users of Mod Organizer. It involves marking merged plugins as hidden (which appends the .mohidden prefix to them) so they are no longer active in your load order. An extension for MO was created to support this workflow with Merge Plugins Standalone.

Pros:
  • Keeps your load order clean and uncluttered.
Cons:
  • Requires a lot of manual steps when rebuilding merges.
  • The MO extension doesn't work with zMerge.
  • Can only be used by Mod Organizer users.

Merge Everything Workflow

Any workflow which involves copying general assets would fall under this designation.

Pros:
  • Creates a single mod package which can be distributed.
Cons:
  • Merges take more time.
  • Rebuilding merges requires re-enabling the mods that were merged unless you use a separate MO profile.
  • Merged mods take up twice as much hard drive space. This is a larger problem if you store your mods on an SSD with limited storage capacity.
  • Cannot be used with mod managers that do not isolate mod data files.
